# 테스트 계획서

## 1. 문서 목적

이 문서는 `AGENTS.md`, `docs/requirements.md`, `docs/data-model.md`, `docs/implementation-plan.md`, `docs/ui`를 기준으로 Zac 서비스의 MVP 구현을 검증하기 위한 테스트 전략과 범위를 정의한다.

테스트의 핵심 목표는 다음과 같다.

- 세계관, 캐릭터, 소속, 관계, 창작물, 챕터가 요구사항대로 동작하는지 검증한다.
- 세계관 권한과 공개 범위가 서버 측에서 일관되게 적용되는지 검증한다.
- 서로 다른 세계관의 리소스가 잘못 연결되지 않도록 데이터 무결성을 검증한다.
- 공개 탐색, 관계 그래프, Markdown 본문, 임시 저장 등 주요 UX가 목업 의도대로 동작하는지 검증한다.
- `docs/requirements.md`의 기능적, 성능적, 보안 요구사항을 추적 가능하게 검증한다.

## 2. 테스트 범위

### 포함 범위

- 회원가입, 로그인, 로그아웃
- 세션 검증과 보호 라우트 접근 제어
- 세계관 CRUD, 목록, 상세, 공개 탐색, 공유 관리
- 캐릭터 CRUD, 목록, 상세, 공개 탐색
- 소속 CRUD, 목록, 상세, 캐릭터-소속 연결
- 관계 CRUD와 그래프 화면
- 창작물 CRUD, 등장 캐릭터/관련 소속 연결, 챕터 편집, 리더
- 태그 입력, 파싱, 생성, 연결 저장
- 공개 범위 계산과 권한별 조회
- 내 작업실, 임시 저장, 선택 삭제
- 페이지네이션, 정렬, 검색
- Markdown 렌더링과 XSS 방어
- 주요 DB 인덱스와 성능 목표

### 제외 범위

MVP 이후 확장 후보는 이번 테스트 계획에서 상세 케이스를 작성하지 않는다.

- 댓글
- 좋아요
- 북마크
- 첨부 파일
- 변경 이력
- 신고/차단
- 작품 공개 예약
- 서버 저장 기반 초안

## 4. 테스트 환경

### 로컬 개발 환경

- Next.js `16.2.6`
- React `19.2.4`
- PostgreSQL
- Prisma `7.8.0`
- NextAuth session 방식

### 테스트 DB

- 개발 DB와 분리된 테스트 DB를 사용한다.
- 테스트는 가능하면 트랜잭션 또는 테스트별 seed/reset으로 독립성을 보장한다.
- seed 데이터는 권한과 공개 범위 조합을 반드시 포함한다.

### 권장 테스트 도구

실제 도구는 구현 단계에서 확정한다.

- 단위/통합 테스트: Vitest 또는 Jest
- E2E 테스트: Playwright
- DB seed/reset: Prisma seed script
- 성능 측정: Playwright trace, Lighthouse, 서버 로그 기반 timing, DB query explain

## 6. 기능 테스트 계획

### 6.1 인증 및 세션

대상 요구사항: `FR-AUTH-001` ~ `FR-AUTH-006`, `SEC-005`

| ID | 테스트 | 기대 결과 |
| --- | --- | --- |
| AUTH-001 | 이메일, 이름, 비밀번호로 회원가입 | 사용자가 생성되고 로그인 가능한 상태가 된다. |
| AUTH-002 | 올바른 계정으로 로그인 | 세션이 생성되고 `/workspace`로 이동한다. |
| AUTH-003 | 잘못된 비밀번호로 로그인 | 로그인 실패 메시지를 표시하고 세션을 만들지 않는다. |
| AUTH-004 | 로그아웃 | 세션이 제거되고 보호 화면 접근 시 로그인으로 안내된다. |
| AUTH-005 | 비로그인 사용자가 쓰기 화면 접근 | 로그인 화면으로 안내된다. |
| AUTH-006 | 서버 쓰기 요청에 세션 없이 직접 접근 | 요청이 거부된다. |

### 6.2 세계관

대상 요구사항: `FR-WORLD-001` ~ `FR-WORLD-009`

| ID | 테스트 | 기대 결과 |
| --- | --- | --- |
| WORLD-001 | 세계관 생성 | 제목, 소개 Markdown, 장르, 커버 이미지, 공개 범위, 태그가 저장된다. |
| WORLD-002 | 세계관 상세 조회 | 접근 가능한 사용자는 상세, 캐릭터, 소속, 창작물 요약을 볼 수 있다. |
| WORLD-003 | 세계관 수정 | 편집 권한 사용자는 수정할 수 있고 변경 내용이 반영된다. |
| WORLD-004 | 세계관 삭제 | 소유자만 삭제할 수 있고 목록에서 제외된다. |
| WORLD-005 | editor가 세계관 삭제 시도 | 권한 없음으로 거부된다. |
| WORLD-006 | 공개 세계관 탐색 | `PUBLIC` 세계관만 노출된다. |
| WORLD-007 | 검색어와 태그 통합 검색 | 제목, 소개, 태그 기준 검색 결과가 반환된다. |
| WORLD-008 | 인기순/최신순 정렬 | `view_count desc`, `updated_at desc` 기준으로 정렬된다. |
| WORLD-009 | 페이지네이션 | 한 페이지 15개씩 반환되고 페이지 이동이 가능하다. |

### 6.3 공유 및 권한

대상 요구사항: `FR-SHARE-001` ~ `FR-SHARE-010`, `SEC-001`, `SEC-003`

| ID | 테스트 | 기대 결과 |
| --- | --- | --- |
| SHARE-001 | 소유자가 이메일로 멤버 공유 | 대상 사용자가 지정 권한으로 멤버 목록에 추가된다. |
| SHARE-002 | 관리자 권한 변경 | OWNER 또는 ADMIN이 멤버 권한을 변경할 수 있다. |
| SHARE-003 | 공유 해제 | 멤버십이 제거되고 대상 사용자는 공유 세계관에 접근할 수 없다. |
| SHARE-004 | VIEWER가 수정 시도 | 모든 쓰기 요청이 거부된다. |
| SHARE-005 | EDITOR가 하위 리소스 생성 | 캐릭터, 소속, 관계, 창작물을 생성할 수 있다. |
| SHARE-006 | outsider가 공유 세계관 조회 | 접근이 거부되고 데이터 존재 여부가 과도하게 노출되지 않는다. |
| SHARE-007 | 목록에 없는 접근 사용자 | 기본적으로 `VIEWER`가 아니라 접근 불가로 처리한다. 공개 리소스만 공개 범위에 따라 조회된다. |

## 14. E2E 핵심 시나리오

### E2E-001: 개인 창작자 기본 흐름

1. 회원가입한다.
2. 로그인한다.
3. 세계관을 생성한다.
4. 소속을 생성한다.
5. 캐릭터를 생성하고 대표 소속을 연결한다.
6. 창작물을 생성하고 캐릭터와 소속을 연결한다.
7. 챕터를 작성한다.
8. 창작물 리더에서 1챕터를 읽는다.

### E2E-002: 공동 창작 권한 흐름

1. OWNER가 세계관을 생성한다.
2. OWNER가 EDITOR를 공유 멤버로 추가한다.
3. EDITOR가 캐릭터와 창작물을 생성한다.
4. VIEWER는 같은 리소스를 조회만 한다.
5. VIEWER가 수정 요청을 보내면 거부된다.
6. OWNER가 공유를 해제하면 EDITOR는 더 이상 접근할 수 없다.

### E2E-003: 공개 탐색 흐름

1. 공개 세계관과 공개 창작물을 준비한다.
2. 비로그인 사용자가 세계관 탐색에서 검색한다.
3. 태그 검색과 인기순/최신순을 변경한다.
4. 카드 클릭으로 상세 화면에 진입한다.
5. 비공개 리소스가 노출되지 않는지 확인한다.

### E2E-004: 관계 그래프 흐름

1. 세계관에 캐릭터 5개와 소속 2개를 만든다.
2. 관계 그래프 화면에서 관계를 추가한다.
3. 간선을 클릭해 관계 수정 모달을 연다.
4. 관계 설명, 상태, 방향성을 수정한다.
5. 관계를 삭제하고 그래프에서 사라지는지 확인한다.

### E2E-005: 임시 저장 복구 흐름

1. 창작물 생성 화면에서 제목과 요약을 입력한다.
2. 저장하지 않고 화면을 벗어난다.
3. 내 작업실 임시 저장 카드에 초안이 표시된다.
4. 카드를 클릭하면 입력값이 복구된다.
5. 저장 후 임시 저장 카드가 제거된다.

## 15. 완료 기준

MVP 테스트 완료는 다음 조건을 만족해야 한다.

- 모든 단위 테스트가 통과한다.
- 모든 통합 테스트가 통과한다.
- E2E 핵심 시나리오가 통과한다.
- 권한 매트릭스 테스트가 통과한다.
- 공개 범위와 비공개 데이터 노출 테스트가 통과한다.
- 주요 성능 목표가 측정 기준 안에 들어온다.
- 알려진 결함은 심각도와 우선순위가 기록되어 있다.
- MVP 완료 기준과 연결된 요구사항에 테스트 결과가 매핑되어 있다.

## 16. 결함 분류

| 심각도 | 기준 |
| --- | --- |
| Critical | 데이터 유실, 비공개 데이터 노출, 인증/권한 우회, 앱 전체 장애 |
| High | 주요 CRUD 불가, 공유/권한 오류, 결제는 없지만 핵심 데이터 불일치 |
| Medium | 특정 화면 또는 일부 입력 흐름 오류, 성능 목표 일부 미달 |
| Low | 문구, 스타일, 경미한 반응형 깨짐, 비핵심 UX 문제 |

Critical 또는 High 결함은 MVP 배포 전 해결해야 한다.
